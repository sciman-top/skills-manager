## Finding 1: Inventory Inherited, Not Verified (MINOR)

**Issue:** The plan's entire inventory of `/mnt/ace/acma-codes` is inherited from the parent plan #2216, which itself was based on pre-session observation due to the same sandbox limitation. This creates a chain of two planning sessions without live verification.

**Severity:** MINOR -- the plan explicitly documents this limitation and requires implementation-time verification.

**Recommendation:** No plan revision needed. The verification requirement is already in the acceptance criteria ("Actual inventory verified against this plan's inherited inventory; deviations documented"). However, the implementation issue should be flagged: if the actual contents differ materially (e.g., 50+ files instead of 6-10), the implementation scope may need to be re-estimated.

**Plan acknowledges this:** Yes, in "Risks" section #1 and #3, and in "Implementation-time verification requirements".

## Finding 2: Exclude Pattern Scope (MINOR)

**Issue:** The plan proposes adding `Thumbs.db`, `desktop.ini`, and `.DS_Store` to the global `exclude_patterns` in config.yaml. While these are sensible, they affect ALL sources, not just acma-codes. This is a global pipeline config change.

**Severity:** MINOR -- the change is net-positive (these artifacts are junk across all sources), but it exceeds the strict #2225 scope of "register and index acma-codes."

**Recommendation:** Accept the change as a sensible collateral improvement. The plan should note that this is a global improvement triggered by acma-codes onboarding, not scoped narrowly to it. No scope violation since these patterns are needed for correct acma-codes indexing and benefit all sources.

**Plan acknowledges this:** Partially -- the plan says "benefits all sources, not just acma-codes" but doesn't flag it as a scope expansion.

## Finding 3: Source Priority Position (APPROVE)

**Issue:** The plan places `acma_codes` after `workspace_spec` and before `api_metadata` in dedup priority. This means acma-codes content is the lowest-priority filesystem source.

**Severity:** None -- this is correct. The acma-codes collection is a staging drop; existing organized sources should take precedence for canonical records. The provenance.py merge logic will preserve the og_standards record as primary and add the acma-codes path as an alias.

**Recommendation:** No change needed.

## Finding 4: Dedup Assessment Output Format Not Specified (MINOR)

**Issue:** The plan describes a dedup assessment (count duplicates vs. new doc_keys) but doesn't specify where the output artifact should be stored or in what format. The "Pseudocode" section says "output summary" but doesn't define the file path.

**Severity:** MINOR -- the implementer will need to decide whether to produce a standalone report file, a GitHub comment summary, or inline console output.

**Recommendation:** Add a note that the dedup assessment should be reported in the implementation PR comment and/or posted as a summary comment on #2225. No need for a permanent artifact file since the dedup results are captured in the index.jsonl provenance arrays.
