### Verdict: REQUEST_CHANGES

### Summary
The plan is directionally strong and aligns with the lean-session objective, but it is missing critical safeguards around data integrity, migration/rollback, and cron reliability. The core simplification is feasible, but removing all Stop-time checks without compensating observability introduces a high risk of silent signal loss. Add explicit failure handling, schema/contracts, and operational acceptance criteria before execution.

### Issues Found
- [P1] Critical: No guaranteed fallback if nightly cron fails or is delayed. Moving all analysis to one batch job creates a single point of failure without defined alerting/SLOs, retry policy, or backfill procedure.
- [P1] Critical: `consume-signals.sh` contract is under-specified. Required JSON schema, invalid input behavior, atomic write/lock behavior, and file rotation/naming collision handling are not defined, risking malformed or lost signals.
- [P2] Important: Acceptance criteria omit functional parity validation. There is no explicit check that data previously emitted by `session-signals.sh`, `engineering-audit.sh`, and `data-provenance.sh` is still captured (directly or via WRK-305 emitters) before their removal.
- [P2] Important: Dependency sequencing is unclear with WRK-305. This plan removes hooks now but depends on “wire signal emitters” later; that can create a capture gap unless ordering is constrained.
- [P2] Important: Scope includes `settings.json` and SKILL docs, but not migration/cleanup of stale state and tooling assumptions (e.g., consumers expecting `accumulator.json` or pending-insights artifacts).
- [P3] Minor: `< 30 lines` is a style constraint, not a reliability criterion; it can conflict with necessary guardrails (validation, safe mkdir, fsync/atomic append, structured logging).

### Suggestions
- Define a strict signal envelope contract (required fields, defaults, version) and add a validation test matrix for empty stdin, malformed JSON, missing session_id, and concurrent writes.
- Add operational criteria: cron heartbeat check, alert on missed nightly run, retry/backfill command, and a rollback switch to temporarily re-enable minimal legacy capture if ingestion drops.
- Gate removal of legacy hooks behind WRK-305 readiness (or include a temporary compatibility adapter) to prevent telemetry blind spots.
- Expand acceptance criteria with end-to-end checks: signal emitted at Stop appears in nightly Phase 1 input and downstream Phase 7/10 outputs within one cron cycle.
- Add explicit non-functional tests: concurrency stress (N parallel Stop events), disk-full/permission error behavior, and path resolution behavior outside git roots.

### Questions for Author
- What is the exact ordering/dependency between WRK-304 and WRK-305 to guarantee no signal-loss window during rollout?
- If nightly `comprehensive-learning` does not run on a given date (host down, cron misfire), what is the expected recovery workflow and who/what detects it?
