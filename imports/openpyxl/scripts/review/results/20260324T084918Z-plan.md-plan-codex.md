### Verdict: REQUEST_CHANGES

### Summary
Current scope is too narrow for the stated acceptance criteria. The proposed `submit-to-gemini.sh` change will not preserve its new diagnostics through the existing cross-review pipeline, and the plan also needs an explicit stderr-redaction story before surfacing auth/capacity errors.

### Issues Found
- [P1] Critical: The plan cannot meet AC-2/AC-3 with a `submit-to-gemini.sh`-only fix. [`cross-review.sh`](/mnt/local-analysis/workspace-hub/scripts/review/cross-review.sh#L400) and [`cross-review.sh`](/mnt/local-analysis/workspace-hub/scripts/review/cross-review.sh#L405) overwrite any non-zero Gemini output with a generic failure stub, then [`cross-review.sh`](/mnt/local-analysis/workspace-hub/scripts/review/cross-review.sh#L422) rewrites unrecognized output to `INVALID_OUTPUT`. On top of that, [`validate-review-output.sh`](/mnt/local-analysis/workspace-hub/scripts/review/validate-review-output.sh#L25) only classifies a narrow set of Gemini `NO_OUTPUT` headings. So the richer stderr diagnostics proposed for [`WRK-5139.md`](/mnt/local-analysis/workspace-hub/.claude/work-queue/pending/WRK-5139.md#L30) would be discarded or normalized away; `cross-review.sh` is not optional here.
- [P2] Important: The plan adds verbatim stderr reporting for auth/capacity failures without a redaction rule. [`review-transport.md`](/mnt/local-analysis/workspace-hub/docs/guides/review-transport.md#L74) states failed provider output is preserved as `.raw.md`, and [`SECURITY.md`](/mnt/local-analysis/workspace-hub/docs/guides/SECURITY.md#L9) forbids committing credentials or internal network information. If Gemini includes account or credential-path details in stderr, this fix would create a new artifact-leak path.

### Suggestions
- Expand the scope to include `cross-review.sh` and `validate-review-output.sh`, and add an end-to-end test proving a Gemini diagnostic failure survives normalization with the expected final artifact text.
- Define a redaction/allowlist policy for stderr before printing it into review artifacts, and cover auth-failure fixtures in tests so sensitive lines never land in `.raw.md` output.

### Questions for Author
- Should the user-visible Gemini result retain the diagnostic text directly, or should only the `.raw.md` sidecar keep it? That choice changes how `cross-review.sh` should normalize failures.
- What stderr fields are explicitly safe to preserve for auth/rate-limit failures, and what must be redacted before artifact preservation?
