## Findings

### Finding 1 (MINOR): Sandbox limitation on live inventory

**What:** The plan's inventory is based on pre-session observation rather than live filesystem traversal. Direct access to `/mnt/ace/acma-codes` was denied by the sandbox.

**Impact:** The inventory may be incomplete. Additional standards families, file types, or nested subdirectories could exist.

**Mitigation in plan:** The plan explicitly documents this limitation in both the inventory section and the risks section. It recommends verification during implementation.

**Verdict:** Acceptable for planning — the known contents are sufficient to design the integration approach. The implementation issues should include a "full inventory verification" step.

### Finding 2 (MINOR): API RP 2SK edition matching ambiguity

**What:** The acma-codes copy is "1996 Dec RP 2SK ... 2nd ed". The transfer ledger has `API-RP-2SK-2ND-ED` as `done`. The wiki references "API RP 2SK (4th Ed, 2024)".

**Analysis:**
- If the acma-codes file is the same binary as the existing `/mnt/ace/0000 O&G/.../` copy → same `doc_key`, link as alias
- If it's a different scan or has different PDF metadata → different `doc_key`, link via `superseded_by` chain
- The wiki page references a much newer edition (4th, 2024) — this is a different document entirely

**Verdict:** The plan correctly handles this via the dedup pass. No change needed, but the implementation should document which scenario actually occurs.

### Finding 3 (MINOR): CSA domain assignment

**What:** CSA Z276 standards cover LNG marine structures and operations. The current taxonomy has 12 domains (structural, marine, pipeline, materials, etc.) with no dedicated LNG domain.

**Analysis:**
- `marine` is the closest fit but is broad (covers mooring, naval architecture, general marine)
- LNG-specific content might benefit from a subdomain tag
- The plan correctly identifies this as a classification question

**Verdict:** Assigning to `marine` domain is pragmatically correct. If CSA Z276 content grows (more LNG standards discovered), a taxonomy expansion request should be filed separately.

### Finding 4 (OBSERVATION): OCIMF 2008 is a predecessor edition

**What:** The acma-codes OCIMF file is "2008 - Mooring Equipment Guidelines" which is the 3rd Edition. The wiki covers MEG4 (4th Ed, 2018).

**Analysis:** These are distinct documents (different content = different `doc_key`). The plan correctly proposes:
- New ledger entry for OCIMF-MEG-2008 (3rd Ed)
- Update to existing `ocimf-meg4.md` wiki page to add historical edition context
- NOT creating a separate wiki page for MEG3 (diminishing returns)

**Verdict:** Approach is sound. The `superseded_by` linkage in the ledger will properly model the edition chain.

### Finding 5 (OBSERVATION): Follow-on issue split is appropriate

**What:** The plan recommends splitting into 4 follow-on issues.

**Analysis:** I tested whether a single implementation issue could work:
- Source registration (modify 1 YAML) — trivial
- Phase A indexing (run 1 script) — operational
- Dedup pass (run provenance.py) — dependent on indexing
- Ledger updates (modify YAML) — dependent on dedup
- Wiki creation (create 3 MD files) — dependent on summaries
- Accessibility update (modify 1 MD) — dependent on wiki

The dependency chain is real and each step has potential blockers (dedup might reveal unexpected results, summaries might need manual review). A single issue would be T3 with poor visibility into progress.

**Verdict:** 4-way split is the right call.
