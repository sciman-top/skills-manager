### Verdict: REQUEST_CHANGES

### Summary
The empty-stdout detection and stderr classification are useful additions, but AUTH_FAILURE is retried when it should bail immediately, classify_stderr is invoked twice redundantly, and _last_stderr_class can report a stale cause from a prior attempt in the final error message.

### Issues Found
- [P2] Design: submit-to-gemini.sh:223-237 — AUTH_FAILURE (401/403/credential errors) is retried with backoff sleep, but auth failures are non-transient and will never self-heal. This wastes up to GEMINI_RETRIES * backoff_seconds before inevitably failing. classify_stderr should return a category, and AUTH_FAILURE should break out of the retry loop immediately.
- [P2] Correctness: submit-to-gemini.sh:255-256 — _last_stderr_class is only set inside the empty-stdout branch (exit_code==0 && raw_size==0). If the final attempt instead fails via the normal path (non-zero exit or render failure at line 240-246), _last_stderr_class retains a stale value from a *previous* empty-stdout attempt, producing a misleading 'cause: CAPACITY_EXHAUSTED' even though the final failure may have been a timeout or render error. Reset _last_stderr_class at the top of each loop iteration, or classify stderr in the normal failure path too.
- [P2] Design: submit-to-gemini.sh:225-226 — classify_stderr is called twice in sequence: once to check the return code (stdout discarded), then again to capture stdout. This greps the file twice unnecessarily. Capture the output once: `_last_stderr_class="$(classify_stderr "$err_file")"` and then check `[[ -n "$_last_stderr_class" ]]`.

### Suggestions
- Add a check after classification: `[[ "$_last_stderr_class" == "AUTH_FAILURE" ]] && break` to avoid retrying non-transient failures.
- Consolidate the double classify_stderr call into a single capture-then-test pattern to avoid redundant file reads.
- Move `_last_stderr_class=""` to the top of the while loop (before run_gemini_once) so it is always fresh per iteration, preventing stale values leaking into the final error message.

### Questions for Author
- Is there a reason AUTH_FAILURE should be retried? If Gemini returns 401/403, retrying seems counterproductive — was there an observed case where auth errors were transient?
- Should the normal failure path (non-zero exit_code or render failure) also classify stderr, so the final error message always reflects the last attempt's cause rather than only empty-stdout attempts?
