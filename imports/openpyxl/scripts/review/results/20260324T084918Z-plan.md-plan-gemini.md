### Summary
The proposed logic to handle empty output and classify errors is conceptually sound. However, a critical implementation detail regarding stderr capture is missing, which is a blocker for the bash script to function as designed.

### Issues Found
- [P1] Critical: The pseudocode relies on inspecting 'stderr' after the command executes. In bash, stderr must be explicitly redirected and captured to a file (e.g., `2> gemini_stderr.log`) during the invocation to be inspected later independently of stdout. This must be explicitly handled to prevent the logic from failing.

### Suggestions
- Modify the design to explicitly capture stderr to a temporary file during the Gemini CLI invocation, and update the string matching logic to grep this file.

### Questions for Author
- How are you planning to capture stderr without mixing it into the stdout pipeline?
- Are we certain that the Gemini CLI writes 'MODEL_CAPACITY_EXHAUSTED' and '429' exclusively to stderr when exit code is 0?
