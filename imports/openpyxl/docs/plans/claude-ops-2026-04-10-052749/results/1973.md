**Module inventory (24 source files):**
The naval_architecture package at `digitalmodel/src/digitalmodel/naval_architecture/` is a mature module. The vessel fleet integration was added to existing files rather than creating isolated new modules, following the established pattern.

**Integration chain verified:**
```
worldenergydata CSV → normalize_fleet_record() → register_fleet_vessels()
                                                       ↓
                                              _SHIPS registry
                                            ↙       ↓         ↘
                              gyradius_for_    hull_hydrostatics()   curves_of_form
                              fleet_vessel()         ↓                   ↓
                                   ↓          stability_curve_     interpolation
                              6-DOF gyradii    estimate()          with fleet fallback
```

**Key design observations:**
- The adapter operates on plain dicts — no direct Python import of worldenergydata. The caller loads the CSV and passes records. This is correct loose coupling.
- The `hull_form_mapper.py` in worldenergydata generates `HULL_LIBRARY_REF` strings for cross-referencing, but this is a string key, not a live import — appropriate for cross-repo boundaries.
- `vessel_type_to_host_type()` maps worldenergydata categories (crane_vessel, drillship, semi_submersible, etc.) to HostType enums used in concept selection.
- The `__init__.py` re-exports all new public functions.

**No issues found.** The implementation follows the existing patterns of the module.
