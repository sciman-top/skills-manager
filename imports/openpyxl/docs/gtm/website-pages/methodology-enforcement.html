Quality assurance that cannot be skipped. Automated gates verify every deliverable before it reaches your team.

Why Instructions Fail

Every engineering organization has quality procedures. Style guides. Review checklists. Design basis documents. The problem is not a lack of written rules -- the problem is that written rules are optional.

A procedure that says "always verify units consistency" is only effective when someone remembers to check. Under deadline pressure, with multiple analyses running in parallel, written procedures are the first thing that gets abbreviated.

The industry pattern: A quality incident occurs. Management responds with a new procedure. The procedure works for three months. Then compliance drifts. Another incident. Another procedure. The stack grows, but the failure rate stays constant.

ACE Engineer takes a fundamentally different approach: we enforce quality rules through automated mechanisms that cannot be bypassed, not through documents that rely on human discipline.

The Enforcement Gradient

Not every rule needs the same enforcement strength. We classify quality controls on a four-level maturity spectrum and systematically promote rules toward stronger enforcement as their importance becomes clear.

Level Mechanism Reliability When We Use It
Level 0 Prose -- Written guidance in skill files Lowest -- only effective if invoked Broad guidance, best practices, contextual recommendations
Level 1 Micro-skill -- Stage-specific checklist, auto-loaded Medium -- guaranteed at stage entry Phase-specific checklists that load automatically when work begins
Level 2 Script -- Automated check, pass/fail High -- auditable and testable Binary verification: units consistent, safety factors met, file formats valid
Level 3 Hook -- Fires automatically on every commit Strongest -- cannot be bypassed Must-never-miss rules: no credentials in deliverables, encoding standards, file size limits

The migration principle: When a prose rule can be expressed as pass/fail, we write a script. When a script must fire on every deliverable, we promote it to a hook. Rules only move toward stronger enforcement -- never weaker.

What This Means for Deliverables

Automated Pre-Delivery Checks

Before any deliverable reaches your team, it passes through automated validation gates. These are not optional review steps -- they are machine-enforced requirements that block delivery until satisfied:

  • Units consistency -- Every calculation is verified for dimensional consistency. Mixed unit systems are flagged automatically.
  • Code compliance references -- Standard clause citations are validated against current editions (DNV, API, ASME, ISO).
  • Input data traceability -- Every design input is linked to its source. Unattributed assumptions are blocked.
  • Calculation convergence -- Numerical results are checked for mesh sensitivity, iteration convergence, and numerical stability.
  • Safety factor verification -- Utilization ratios and safety factors are automatically compared against code-required minimums.

Review Gate Enforcement

Our multi-perspective review process (see Compound Engineering) is not advisory. Review findings are classified by severity:

  • Critical -- Blocks delivery. Must be resolved and re-verified.
  • Major -- Blocks delivery. Requires documented resolution.
  • Minor -- Does not block delivery but is tracked and resolved before project close.
  • Observation -- Logged as knowledge for future projects.

A deliverable with any open Critical or Major finding cannot be transmitted to the client. The enforcement mechanism makes this a technical impossibility, not a management decision.

The Engineering Case for Enforcement

Human Discipline Has Known Failure Modes

Even experienced engineers skip steps under pressure. This is not a character flaw -- it is a well-documented cognitive pattern. Checklists work when workload is manageable. Under deadline pressure, humans optimize for speed and unconsciously skip verification steps.

Automated enforcement removes this failure mode entirely. The gate does not know about the deadline. It does not optimize for speed. It checks every condition, every time.

Enforcement Creates Auditability

Every automated check produces a timestamped, immutable record. When a client or regulator asks "how did you verify this?", the answer is not "our engineer followed the procedure." The answer is: "Here is the automated verification log showing every check that passed, when it passed, and what was tested."

Enforcement Scales Without Degradation

A procedure-based quality system degrades as the team scales. More people means more variation in how procedures are interpreted and followed. Enforcement-based quality maintains the same standard regardless of team size -- the automated gate applies identically to every deliverable.

Practical Examples

Example: Wall Thickness Calculation

A pipeline wall thickness calculation per DNV-ST-F101 involves safety class selection, load combinations, material de-rating, and corrosion allowance. A prose-level rule says "verify safety class selection." Our enforcement approach:

  1. Level 2 script validates that the safety class matches the fluid category and location class from the design basis
  2. Level 2 script verifies that the temperature de-rating factor corresponds to the maximum design temperature
  3. Level 3 hook blocks any deliverable where the calculated utilization exceeds the code-allowed maximum

The engineer cannot accidentally use the wrong safety class. The system catches it before the calculation proceeds.

Example: Environmental Data Provenance

Metocean data drives every offshore design. Using outdated hindcast data or incorrect return periods invalidates the entire analysis. Our enforcement:

  1. Level 1 micro-skill prompts for data source, vintage, and return period at the start of every environmental loading phase
  2. Level 2 script validates that referenced metocean files match the project's geographic coordinates within tolerance
  3. Level 3 hook blocks delivery if any environmental load case references data older than the project-specified vintage
