## What the Plan Defines

The plan establishes a formal intelligence retrieval contract for GitHub issue workflows, covering:

1. **Five workflow stages** (intake, planning, execution, review, closeout) with minimum retrieval requirements at each stage.

2. **Six issue classes** (general, engineering, data-pipeline, documentation, harness/infrastructure, knowledge/intelligence) with class-specific retrieval bundles listing concrete file paths.

3. **Evidence placement rules** — exactly where retrieval evidence must appear:
   - In plan files: tightened Resource Intelligence Summary with ≥3 sources and 5 required sub-sections
   - In review artifacts: new Retrieval Adequacy section with 5-row check table and verdict
   - In GitHub comments: "Sources consulted" at plan-review, "Sources consumed" + "Promotion candidates" at closeout

4. **Measurable checks** — 10 named checks (RC-1 through RC-5 for plans, RR-1 through RR-3 for reviews, RCL-1 through RCL-2 for closeout) with pass conditions and enforcement levels.

5. **Progressive adoption** of #2104 (entry points) and #2136 (accessibility registry) — the contract works with current file paths today and improves as siblings deliver.

## Quality of the Contract

### Strengths

1. **Grounded in real artifacts.** Every retrieval bundle source is a concrete file path that exists in the repo today. No abstract "consult the knowledge base" — agents can follow the paths.

2. **Measurable.** The 10 named checks (RC/RR/RCL) have binary pass conditions. An auditor can read a plan and verify compliance without subjective judgment.

3. **Progressive.** The contract degrades gracefully when #2104/#2136 don't exist yet and improves automatically when they land. No blocking dependency.

4. **Layered enforcement.** Each check has an enforcement level (template → convention → linter → script → hook). This matches the repo's enforcement gradient (`.claude/rules/patterns.md`).

5. **Practical minimum.** The ≥3 source floor is low enough for T1 trivial issues (issue body + code search + prior plans) but creates a real bar against empty Resource Intelligence Summaries.

### Weaknesses (non-blocking)

1. **No automated enforcement yet.** The contract is a specification, not an enforced gate. Until linters or hooks exist, compliance depends on reviewer discipline.

2. **Close-comment checks require GitHub API.** Unlike plan-file and review-artifact checks (file-system readable), the closeout checks require querying GitHub issue comments. This is harder to automate but acceptable for now.

3. **Issue-class classification relies on labels.** If labels are missing, the default-to-General rule applies, which means the minimum retrieval bundle is the weakest. This is correct behavior (no data → conservative retrieval) but means unlabeled issues get less retrieval discipline.
