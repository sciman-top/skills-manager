### Verdict: REQUEST_CHANGES

### Summary
The plan has a command-injection risk in the GH→local sync path and a false atomicity claim in the highest-risk migration script. Both must be addressed before execution begins.

### Issues Found
- [P1] Critical — Command injection via gh-sync-down.sh: GitHub Issue titles, labels, and body content are attacker-controlled input. The plan has this data written into local YAML/markdown files that are subsequently parsed by shell scripts (whats-next.sh, rebuild-wrk-index.sh, etc.) using grep/sed/awk. Without explicit input sanitization (at minimum: stripping shell metacharacters, quoting all expansions, rejecting embedded newlines in single-line fields), a crafted issue title like `$(rm -rf .)` or a YAML injection payload can execute arbitrary commands or corrupt the work queue when the local scripts source/parse those files. The plan must specify a sanitization strategy for all GH-sourced fields before they touch the local filesystem.
- [P1] Critical — False atomicity claim in renumber-to-gh.sh: The plan states 'Uses git mv for atomicity' but git mv in a loop is NOT atomic — if the script fails mid-iteration (disk full, name collision, signal), the working tree is left in a partially-renamed state with cross-references half-updated. The 'git commit checkpoint before' mitigation is recovery (git reset --hard), not prevention. For a self-described 'highest-risk operation' at ~150 lines, the plan must either (a) implement a two-phase approach (compute full rename map + validate first, apply all renames second) with an explicit rollback script, or (b) use a transactional strategy (rename into a temp directory tree, validate, then swap).
- [P1] Critical — No explicit rollback procedure for Phase D: The plan mentions a 'git commit checkpoint' but does not document the actual rollback command sequence, who decides to rollback, or how to detect that a partial failure occurred. Given that renumbering touches every WRK file, cross-references, and potentially hooks/skills, a one-line mention is insufficient. A documented, tested rollback script or procedure is required.

### Suggestions
- Add a sanitize_gh_field() function to gh-sync-down.sh that strips or escapes shell metacharacters ($, `, \, ", ', newlines) from all GH API output before writing to local files. Consider using jq --raw-output with explicit allowlists for field characters rather than shell variable expansion of API responses.
- Redesign renumber-to-gh.sh as a two-phase script: Phase 1 computes the full rename map, validates no collisions, and writes a manifest file. Phase 2 reads the manifest and applies renames. Include an undo-renumber.sh script that reads the same manifest in reverse. Gate Phase 2 on Phase 1 succeeding completely.
- Add a rollback section to the plan with explicit commands: what to run, how to detect partial failure (e.g., manifest shows N renames but only M completed), and a smoke test to confirm rollback succeeded.

### Questions for Author
- How are GitHub API responses consumed in gh-sync-down.sh — via gh issue list --json piped to jq, or via shell variable expansion? This determines the injection surface area.
- Has renumber-to-gh.sh been prototyped? Are there known filename collisions between WRK-IDs and GH issue numbers that could cause git mv failures mid-batch?
