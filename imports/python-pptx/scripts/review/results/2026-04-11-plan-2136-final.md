## Key Design Decisions

1. **Federation over consolidation.** The registry is a separate YAML file at `data/document-index/intelligence-accessibility-registry.yaml`, not a merge with existing registries. Each registry serves a distinct abstraction level: `index.jsonl` tracks documents, `standards-transfer-ledger.yaml` tracks standards, `mounted-source-registry.yaml` tracks source roots, and the new registry tracks intelligence surfaces.

2. **`asset_key` complements `doc_key`.** Individual documents use `doc_key` (SHA-256) per #2207. Intelligence surfaces (directories, collections, regularly-updated YAML files) use `asset_key` (human-readable slug). The two identity systems are complementary, not competing.

3. **Seeded from evidence, not speculation.** The initial ~26 entries come directly from the #2096 accessibility map inventory (Section 5). No assets are invented; every entry maps to an existing repo artifact.

4. **Minimal implementation surface.** Only 4 files to create/modify: the registry YAML, a validation script, a weekly review template update, and this plan's index row. No CLI tools, no API, no conformance checks — those belong to future issues.
