## MINOR Findings

### M1: `asset_key` naming convention could use a formal grammar

**Finding:** The `<type>-<short-name>` convention is described in prose but not specified as a formal pattern (e.g., regex). This could lead to inconsistency as entries are added.

**Recommendation:** Add a regex pattern to the validation expectations: `^[a-z][a-z0-9]*(-[a-z0-9]+)*$`. This is not blocking — the convention is clear enough for initial seeding.

**Plan revision:** Added `test_asset_key_format` concept to validation expectations (noted in validation section: "asset_key matches pattern: lowercase-alphanumeric-with-hyphens").

### M2: `query_command` as a shell string is fragile

**Finding:** The plan recommends starting with shell command strings (e.g., `yq '.codes[]' data/design-codes/code-registry.yaml`) for simplicity. These are machine-specific, shell-dependent, and may break across environments.

**Recommendation:** Accept the simplicity trade-off for v1 but document the intention to migrate to a structured format if usage grows. This is already noted in Open Questions #5.

**Plan revision:** None needed — the open question already addresses this.

### M3: No explicit versioning strategy for the registry schema

**Finding:** The plan includes `schema_version: '1.0.0'` in the pseudocode but does not define what constitutes a breaking change or how schema migrations would work.

**Recommendation:** For v1, the `schema_version` field is sufficient. Define migration rules only when a v2 is needed. This is appropriate for a new file with no consumers yet.

**Plan revision:** None needed — the risk is low for initial implementation.

### M4: Coverage test against #2096 could be more specific

**Finding:** `test_coverage_vs_2096_inventory` is described as "compare asset list against #2096 Section 5 inventory" but doesn't specify the expected count or list of required assets.

**Recommendation:** The test should enumerate the ~26 assets from #2096 Sections 5.1-5.4 and verify each has a corresponding `asset_key` entry. This makes the test deterministic.

**Plan revision:** The seeding strategy section already enumerates the specific assets. The test should reference that enumeration.

### M5: `machine_scope` controlled vocabulary not defined

**Finding:** The plan says machine names should come from "a controlled vocabulary matching `mounted-source-registry.yaml` source_ids or known machine hostnames" but does not define the initial vocabulary.

**Recommendation:** Define the initial vocabulary explicitly:
- `all-git-clones` — available on any machine with a repo clone
- `ace-linux-1` — primary workstation with local mounts
- `ace-linux-2` — NFS-mounted secondary
- `dev-secondary` — DDE remote drive

This vocabulary should be documented in the registry file header.

**Plan revision:** The plan's Open Question #3 partially addresses this. The vocabulary can be finalized during implementation.
