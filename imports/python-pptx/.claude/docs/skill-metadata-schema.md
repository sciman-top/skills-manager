# SKILL.md Metadata Schema
*Issue: #1481 | Created: 2026-03-29*

---

## Frontmatter Fields

### Required

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Skill identifier (kebab-case, matches directory name) |
| `description` | string | One-line action-oriented description. Start with a verb or "Use when..." |
| `version` | string | Semantic version (`major.minor.patch`) |
| `category` | string | Top-level domain folder name (e.g., `development`, `data`, `ai`) |

### Recommended Optional

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `tags` | string[] | `[]` | Discovery keywords — domain topics, tool names, use-case labels. Used by INDEX.md and skill search. |
| `related_skills` | string[] | `[]` | Companion skill names (kebab-case) that are often used alongside this skill or share a workflow. |

### Other Optional

| Field | Type | Description |
|-------|------|-------------|
| `last_updated` | string | ISO date of last meaningful content change |
| `source` | string | URL if adapted from an external repo |
| `capabilities` | string[] | Machine-readable capability labels |
| `requires` | string[] | Runtime dependencies (files, tools, services) |
| `triggers` | string[] | Natural-language phrases that should activate the skill |
| `see_also` | string[] | Sub-skill IDs (auto-managed by skill tooling) |
| `scripts_exempt` | boolean | Skip script-tier quality check |
| `deprecated` | boolean | Marks skill as retired |
| `deprecated_date` | string | ISO date of deprecation |
| `replacement` | string | Name of successor skill |
| `official_plugin` | string | Plugin marketplace identifier |

## Tags Guidelines

Tags are **flat, lowercase, kebab-case** strings. They describe *what* the skill is about, not how it works.

**Good tags**: `code-review`, `python`, `security`, `excel`, `quota-management`, `debugging`
**Bad tags**: `useful`, `important`, `v2`, `WIP`

**Categories to draw from**:
- **Domain**: `marine-engineering`, `finance`, `legal`, `data-science`
- **Language/tool**: `python`, `typescript`, `excel`, `docker`, `git`
- **Activity**: `code-review`, `debugging`, `testing`, `documentation`, `automation`
- **Concern**: `security`, `performance`, `compliance`, `quality`

Aim for 3-7 tags per skill. Avoid duplicating the `category` field — tags add specificity.

---

## Related Skills Guidelines

List skill names (not paths) of skills that:
- Are commonly invoked in the same workflow
- Provide complementary capability (e.g., `tdd-obra` + `systematic-debugging`)
- Share domain context (e.g., `dark-intelligence-workflow` + `legal-sanity-scan`)

Order by relevance. Keep to 2-5 entries.

## Hermes Compatibility

Hermes Agent nests tags under `metadata.hermes.tags`. Our schema keeps `tags` and `related_skills` at the top level for simplicity and backward compatibility. When importing Hermes skills, flatten the nested fields:

```yaml
# Hermes format
metadata:
  hermes:
    tags: [debugging, troubleshooting]
    related_skills: [tdd]

# Workspace-hub format (flatten)
tags: [debugging, troubleshooting]
related_skills: [tdd]
```

## INDEX.md Surfacing

Category-level `INDEX.md` files auto-generated by tooling include a **Tags** column. When `tags: []` is empty, the column shows `—`. Populating tags in SKILL.md frontmatter will automatically surface them in the index table on next regeneration.
